# 原理知识

本文档整理当前项目中实际用到的算法、图论概念和数学原理。

当前项目不是机器学习模型，也不是深度学习模型，而是一个基于图论和基础统计的规则识别系统。

## 一、整体算法组合

当前项目主要使用了以下算法和思想：

```text
二部图建模
邻接表存储
二跳邻居搜索
集合去重
二部图单侧投影
加权图
连通分量
用户同构图投影
Louvain 社区发现
服务端标准 Leiden 社区发现
离线快照 Leiden-like 社区细化
网络密度
度中心性
加权度
介数中心性
PageRank
属性共享密度
比例统计
线性加权规则评分
```

一句话概括：

```text
二部图 + 二跳邻居搜索 + 中介投影图 + 用户同构图 + 连通分量 / Louvain / Leiden + 中心性指标 + 规则风险评分
```

## 二、图论建模

将业务数据抽象为一张图：

```text
G = (V, E)
```

其中：

```text
V = 节点集合
E = 边集合
```

在当前业务中：

```text
V = 中介节点集合 ∪ 借款人节点集合
E = 中介和借款人的关联关系
```

因为边只连接「中介」和「借款人」，所以它是一张二部图：

```text
G = (U, W, E)
```

其中：

```text
U = 中介集合
W = 借款人集合
E ⊆ U × W
```

业务含义：

```text
收货人手机号 -> 中介节点
借款人手机号 -> 借款人节点
一行贷款记录 -> 一条中介和借款人的关联记录
```

## 三、邻接表存储

项目没有使用邻接矩阵，而是使用邻接表。

核心结构：

```python
agent_to_borrowers
borrower_to_agents
loans_by_borrower
```

数学含义：

```text
agent_to_borrowers[a] = 与中介 a 相连的借款人集合
borrower_to_agents[b] = 与借款人 b 相连的中介集合
loans_by_borrower[b] = 借款人 b 的贷款记录集合
```

邻接表适合稀疏图。风控关系图通常节点多，但任意两个节点之间并不都会相连，因此邻接表比邻接矩阵更节省空间。

## 四、一度邻居与二度邻居

对任意节点 `v`，一度邻居定义为：

```text
N(v) = {u | (v, u) ∈ E}
```

二度邻居定义为：

```text
N2(v) = ⋃ N(u), u ∈ N(v)
```

然后去掉自身：

```text
N2(v) = N2(v) - {v}
```

## 五、二度中介数

对某个中介 `a`：

```text
N(a) = 该中介的一度借款人集合
```

二度中介集合：

```text
S_agent(a) = ⋃ Agents(b), b ∈ Borrowers(a)
```

去掉自己：

```text
S_agent(a) = S_agent(a) - {a}
```

二度中介数：

```text
second_degree_agents(a) = |S_agent(a)|
```

其中 `|S|` 表示集合 `S` 的元素个数。

## 六、二度借款人数

对某个借款人 `b`：

```text
N(b) = 该借款人的一度中介集合
```

二度借款人集合：

```text
S_borrower(b) = ⋃ Borrowers(a), a ∈ Agents(b)
```

去掉自己：

```text
S_borrower(b) = S_borrower(b) - {b}
```

二度借款人数：

```text
second_degree_borrowers(b) = |S_borrower(b)|
```

## 七、集合去重

二度节点可能通过多条路径到达。

例如：

```text
借款人 B1 -> 中介 A1 -> 借款人 B2
借款人 B1 -> 中介 A2 -> 借款人 B2
```

借款人 `B2` 被两条路径命中，但最终只能算一次。

因此项目使用集合运算：

```text
set union
set difference
set cardinality
```

也就是：

```text
并集
差集
集合基数
```

## 八、二部图单侧投影

为了识别中介团伙，需要把「中介 - 借款人」二部图投影成「中介 - 中介」图。

设二部图邻接矩阵为：

```text
B
```

其中：

```text
B[i, j] = 1 表示中介 i 关联借款人 j
B[i, j] = 0 表示无关联
```

投影到中介侧：

```text
P = B × Bᵀ
```

其中：

```text
P[i, k] = 中介 i 和中介 k 共享的借款人数
```

当：

```text
P[i, k] > 0
```

表示两个中介共享过借款人，因此在中介投影图中连边。

对角线：

```text
P[i, i]
```

表示中介 `i` 自己关联的借款人数。实际构图时会忽略对角线。

## 九、加权中介投影图

中介投影图可以表示为：

```text
G_agent = (A, E_agent, w)
```

其中：

```text
A = 中介节点集合
E_agent = 中介之间的共享关系边集合
w(i, j) = 中介 i 和中介 j 的共享关系强度
```

权重越大，说明两个中介之间的共享关系越强。

当前项目保留三种中介投影口径。

第一种是原始的「共享用户」口径：

```text
Agent -- 共享用户 -- Agent
```

也就是两个中介至少共享 1 个借款人，就认为它们存在关联：

```text
w_user(i, j) = 中介 i 和中介 j 共享的借款人数
```

第二种是「连通分量关系」口径，用于默认的疑似团伙识别。它同时比较两个中介的共享借款人，以及两个中介各自名下直接申请记录中的共享属性：

```text
shared_user(i, j) = 两个中介共同关联过的借款人数
shared_device(i, j) = 中介 i 名下直接申请记录的设备集合，与中介 j 名下直接申请记录的设备集合的交集数量
shared_ip(i, j) = 中介 i 名下直接申请记录的 IP 集合，与中介 j 名下直接申请记录的 IP 集合的交集数量，其中只统计满足当前 IP人数阈值的 IP
shared_addr(i, j) = 中介 i 名下直接申请记录的地址簇集合，与中介 j 名下直接申请记录的地址簇集合的交集数量
```

其中 `IP人数` 是该 IP 关联过的去重借款人数。共享设备、共享 IP、共享地址簇不要求两个中介必须共享借款人；它们只使用两个中介各自名下直接申请记录中的属性集合，也不会把借款人在第三个中介名下的设备、IP、地址簇反算进这两个中介的共享关系。

过滤连边规则会根据首复贷筛选状态变化。

如果首复贷选择的是某一个具体取值，例如仅首贷或仅复贷：

```text
shared_device >= 1
OR shared_user >= 1
OR (shared_ip >= 2 AND IP人数 <= 20)
OR shared_addr >= 1
```

如果首复贷选择的是全部数据：

```text
shared_device >= 1
OR shared_user >= 2
OR (shared_ip >= 2 AND IP人数 <= 50)
OR shared_addr >= 1
```

第三种是「多关系」口径：

```text
Agent -- 共享用户 -- Agent
Agent -- 共享设备 -- Agent
Agent -- 共享 IP -- Agent
Agent -- 共享地址簇 -- Agent
```

当前多关系边权为：

```text
w_multi(i, j)
= 1.0 * 共享用户数
+ 1.2 * 共享设备数
+ 0.8 * 共享 IP 数
+ 1.0 * 共享地址簇数
```

因此两个中介即使没有共享用户，只要共享设备、IP 或地址簇，也可能在多关系中介投影图中连边；共享用户刻画人群交叉，设备、IP、地址簇刻画资源复用。

后续可以调高阈值：

```text
w(i, j) >= 2
w(i, j) >= 3
```

这样可以过滤弱关系，减少误报。

特征宽表中：

```text
节点度数(共享用户)
加权度数(共享用户)
PageRank分数(共享用户)
中介中心性(共享用户)
连通子图大小(共享用户)
```

来自原始共享用户中介投影图。

```text
节点度数(连通分量关系)
加权度数(连通分量关系)
PageRank分数(连通分量关系)
中介中心性(连通分量关系)
连通子图大小(连通分量关系)
```

来自过滤后的连通分量关系图，也是默认连通分量社区字段的来源。

```text
节点度数(多关系)
加权度数(多关系)
PageRank分数(多关系)
中介中心性(多关系)
连通子图大小(多关系)
```

来自新增多关系中介投影图。

另外：

```text
节点度数(共享设备) / 加权度数(共享设备)
节点度数(共享IP) / 加权度数(共享IP)
节点度数(共享地址簇) / 加权度数(共享地址簇)
```

用于拆解多关系中每类证据各自贡献了多少连接。

## 十、连通分量与疑似团伙

在图中，如果节点 `u` 和节点 `v` 之间存在路径：

```text
u -> ... -> v
```

则称 `u` 和 `v` 连通。

连通分量是图中的极大连通子图。

当前项目将过滤后的中介关系图中的连通分量定义为疑似中介团伙：

```text
community = connected_component(G_filtered_agent)
```

例如：

```text
A -- B -- C
```

即使 `A` 和 `C` 没有直接连边，只要它们通过 `B` 连通，也会被归入同一个疑似团伙。

该方法的优点：

```text
简单
可解释
速度快
适合原型阶段
```

该方法的局限：

```text
只要链路连通就会合并，可能把弱相关群体并到一起
无法自动识别重叠社区
不区分强边和弱边对社区边界的影响
```

## 十一、同构图与 Louvain / Leiden

除了连通分量，当前项目还支持把多关系图投影成同构图，再做社区发现。页面下拉框仍然只有 Louvain / Leiden 两个选项；通用社区字段使用用户同构图结果，宽表同时额外输出 Agent 后缀字段。

### 用户同构图

原始多关系可以表示为：

```text
User - Agent
User - Device
User - IP
User - AddressCluster
```

投影后的用户图可以表示为：

```text
G_user = (B, E_user, w)
```

其中：

```text
B = 借款人 / 用户节点集合
E_user = 用户之间的共享关系边集合
w(i, j) = 用户 i 和用户 j 共享关系的加权强度
```

当前权重规则：

```text
共享中介：+1.0
共享设备：+1.2
共享 IP：+0.8
共享地址簇：+1.0
```

### 中介同构图

当选择 Louvain / Leiden 时，宽表还会同步生成 Agent 版社区字段。它使用未过滤的 Agent-Agent 多关系加权图，不使用连通分量过滤规则。

```text
Agent -- 共享用户 -- Agent
Agent -- 共享设备 -- Agent
Agent -- 共享 IP -- Agent
Agent -- 共享地址簇 -- Agent
```

这里的共享设备、共享 IP、共享地址簇同样只比较两个中介各自名下直接申请记录中的属性集合，不以共同借款人为前提。

投影后的中介图可以表示为：

```text
G_agent_multi = (A, E_agent_multi, w)
```

权重规则为：

```text
w(i, j)
= 1.0 * 共享用户数
+ 1.2 * 共享设备数
+ 0.8 * 共享 IP 数
+ 1.0 * 共享地址簇数
```

然后在 `G_agent_multi` 上运行 Louvain / Leiden，生成：

```text
社区id(Agent)
社区规模(Agent)
社区借款人数(Agent)
社区坏账率(Agent)
节点度数(Agent)
PageRank分数(Agent)
中介中心性(Agent)
```

权重越高，表示两个用户之间的关系证据越强。

Louvain 的目标是最大化模块度：

```text
Q = 社区内部边权占比 - 随机图下的期望内部边权占比
```

直观理解：

```text
社区内部连接越密，社区之间连接越稀疏，模块度越高。
```

服务版 `server.py` 使用工业图库实现：

```text
Louvain：igraph.community_multilevel
Leiden：leidenalg.find_partition，使用 RBConfigurationVertexPartition
```

其中 Louvain 通过多轮局部移动和图聚合来最大化模块度；Leiden 在 Louvain 思路上增加 refinement，避免社区内部连接质量差或不连通的问题，更适合生产图社区发现。

离线快照 `snapshot.html` 为了保持单文件可打开，仍保留轻量 Louvain / Leiden-like 实现。也就是说，工业版 Louvain / Leiden 只在服务端页面生效。

## 十二、组合数学与网络密度

如果一个团伙有 `n` 个中介，则理论最大边数是：

```text
C(n, 2) = n(n - 1) / 2
```

其中 `C(n, 2)` 是组合数，表示从 `n` 个节点中任选 2 个节点形成一条边。

设实际边数为：

```text
m
```

网络密度为：

```text
density = m / C(n, 2)
```

密度越接近 1，说明团伙内部越接近完全图，关系越紧密。

## 十三、度与度中心性

节点 `v` 的度定义为：

```text
deg(v) = |N(v)|
```

在中介投影图中，`deg(v)` 表示该中介直接连接了多少其他中介。

度中心性定义为：

```text
C_D(v) = deg(v) / (n - 1)
```

其中：

```text
n = 团伙内中介节点数
```

度中心性越高，说明该中介在团伙内的直接连接越多。

## 十四、加权度

普通度只看连接数量，不看连接强度。

加权度考虑边权：

```text
deg_w(v) = Σ w(v, u), u ∈ N(v)
```

在当前业务中：

```text
deg_w(v) = 该中介与其他中介共享借款人数之和
```

加权度越高，说明该中介和其他中介之间的共享客户关系越强。

## 十五、介数中心性

介数中心性衡量一个节点是否位于其他节点之间的最短路径上。

对节点 `v`：

```text
C_B(v) = Σ σ_st(v) / σ_st
```

其中：

```text
s ≠ v ≠ t
σ_st = s 到 t 的最短路径总数
σ_st(v) = s 到 t 的最短路径中经过 v 的路径数
```

如果很多节点对之间的最短路径都经过某个中介，则该中介的介数中心性高。

业务含义：

```text
该中介可能是多个中介圈层之间的桥接节点。
```

项目中的实现接近 Brandes algorithm 的无权图版本：

```text
1. 对每个源点做 BFS
2. 统计源点到其他节点的最短路径数量
3. 反向累积依赖值
4. 得到每个节点的介数中心性
```

为了避免大团伙计算过慢，当前对特别大的社区会跳过精确介数中心性计算。

## 十六、PageRank

PageRank 用于衡量中介在整体中介投影图中的全局重要性。

它的核心思想是：

```text
一个中介的重要性，不只取决于它连接了多少中介，
还取决于这些相连中介本身是否重要。
```

数学上可以写成迭代形式：

```text
PR(v) = (1 - d) / N + d * Dangling / N + d * Σ PR(u) * w(u, v) / Σ w(u, x)
```

其中：

```text
d = 阻尼系数，当前使用 0.85
N = 中介节点总数
Dangling = 当前轮所有无邻居节点的 PageRank 分数之和
w(u, v) = 中介 u 和中介 v 的共享借款人数
Σ w(u, x) = 中介 u 对外连接权重之和
```

在当前项目中，PageRank 是在「中介投影图」上计算的，并且使用共享借款人数作为边权。

业务含义：

```text
PageRank 高的中介，可能处在全局关系网络中更核心的位置。
```

它和度中心性的区别：

```text
度中心性偏局部，看直接连接数量。
PageRank 偏全局，看连接对象的重要性。
```

### Dangling 节点的递推过程

如果某个中介在共享用户图里没有任何邻居，也就是：

```text
adjacency(v) = {}
```

那么它是一个 dangling 节点。

这类节点没有任何邻居传入项，因此它的迭代公式会退化为：

```text
PR(v) = (1 - d) / N + d * Dangling / N
```

假设一共有 `D` 个 dangling 节点，并且它们在初始时分数相同、图结构也相同，那么每个 dangling 节点在每一轮的分数都相同。设这个共同分数为 `r`，则：

```text
Dangling = D * r
```

代回公式可得：

```text
r_next = (1 - d) / N + d * D * r / N
```

这就是 dangling 节点的递推关系。

### 当前项目中的一个具体例子

以当前项目的一份样本数据为例：

```text
N = 912
D = 671
d = 0.85
```

于是：

```text
(1 - d) / N = 0.15 / 912 = 0.0001644736842105263
d * D / N = 0.85 * 671 / 912 = 0.6253837719298246
```

所以 dangling 节点的递推式为：

```text
r_next = 0.0001644736842105263 + 0.6253837719298246 * r
```

PageRank 初始化时，每个节点分数都相同：

```text
r_0 = 1 / 912 = 0.0010964912280701754
Dangling_0 = 671 * r_0 = 0.7357456140350878
```

随后不断迭代：

```text
r_1 = 0.0001644736842105263 + 0.6253837719298246 * r_0
    = 0.0008502015043090181
Dangling_1 = 671 * r_1 = 0.5704852093913512

r_2 = 0.0001644736842105263 + 0.6253837719298246 * r_1
    = 0.0006961759078757111
Dangling_2 = 671 * r_2 = 0.46713403418460214

r_3 = 0.0005998507994045086
Dangling_3 = 0.4024998864004253

r_4 = 0.0005396106397372385
Dangling_4 = 0.362078739263687
```

继续迭代 40 轮后，分数会收敛到：

```text
r ≈ 0.00043904581172278317
Dangling = 671 * r ≈ 0.29459973966598335
```

也就是说，`Dangling` 不是额外定义的业务指标，而是：

```text
当前轮所有孤立中介节点 PageRank 分数的总和
```

当图里有大量孤立节点时，这个量会在 PageRank 中形成一个“均摊回流项”，平均分配给所有节点。

## 十七、属性共享密度

新增设备、IP、地址簇后，可以把这些属性也看成中介之间的共享媒介。

例如两个中介使用过同一个设备：

```text
中介 A -> 设备 D <- 中介 B
```

则中介 A 和中介 B 在设备维度上形成一条共享边。

社区设备密度：

```text
device_density = 团伙内共享设备形成的中介边数 / C(n, 2)
```

社区 IP 密度：

```text
ip_density = 团伙内共享 IP 形成的中介边数 / C(n, 2)
```

社区地址密度：

```text
address_density = 团伙内共享地址簇形成的中介边数 / C(n, 2)
```

这些指标用于衡量团伙是否不仅共享借款人，还共享设备、IP 或地址簇。

## 十八、比例统计

逾期率和坏账率都是比例统计。

基本形式：

```text
rate = overdue_count / total_count
```

当前业务口径中，只有：

```text
final_result = 30
```

才表示借款通过 / 已放款。因此坏账率、逾期率、贷款笔数等贷款口径指标，只统计 `final_result=30` 的样本。

未通过样本仍然保留在图谱关系和申请笔数中，用于观察申请关系网络，但不会进入坏账率分母。

二度借款逾期率：

```text
second_degree_overdue_rate =
二度借款人逾期贷款笔数 / 二度借款人总贷款笔数
```

团伙坏账率：

```text
bad_debt_rate =
团伙覆盖借款人中 final_result=30 的逾期贷款笔数 / 团伙覆盖借款人中 final_result=30 的总贷款笔数
```

当前口径是按贷款笔数计算，不是按人数计算。

## 十九、线性加权规则评分

当前 `risk_score` 是解释性规则分，不是训练得到的机器学习分数。

数学形式是线性组合：

```text
S = Σ wi * xi
```

展开后可以理解为：

```text
risk_score =
w1 * bad_debt_rate
+ w2 * density
+ w3 * shared_ratio
+ w4 * community_size_factor
```

其中：

```text
bad_debt_rate = 团伙坏账率
density = 团伙网络密度
shared_ratio = 共享借款人数 / 团伙覆盖借款人数
community_size_factor = 团伙规模因子
```

这些因素越高，风险分越高。

当前规则分的作用：

```text
排序
辅助筛查
解释风险来源
```

它不代表真实违约概率。

## 二十、当前没有使用的算法

当前服务端已经支持连通分量、igraph Louvain 和 leidenalg Leiden 社区划分，但还没有使用以下算法：

```text
Label Propagation
Node2Vec
GraphSAGE
GNN
监督学习分类模型
无监督异常检测模型
```

如果后续数据规模变大，或者需要更精细地划分团伙，可以考虑引入：

```text
Neo4j GDS
NetworkX
igraph
图嵌入
异常检测模型
```

## 二十一、当前方法的适用边界

当前方法适合：

```text
快速原型
小到中等规模数据
关系可解释分析
疑似团伙初筛
规则风控特征生成
```

需要谨慎的地方：

```text
连通分量可能过度合并弱关系
共享一个借款人不一定代表真实团伙
坏账率在样本量很小时波动较大
规则风险分不是概率
中心性指标会受团伙规模影响
```

因此实际使用时，建议结合：

```text
最小共享借款人数阈值
最小团伙规模阈值
最小贷款笔数阈值
时间窗口
渠道信息
人工复核
```

## 二十二、后续可升级方向

可以进一步引入更强的图算法：

```text
Louvain / Leiden：更细粒度社区发现
PageRank：识别全局影响力中介
k-core：识别核心团伙骨架
最短路径分析：识别跨团伙连接路径
图嵌入：把节点关系转成模型可用向量
异常检测：识别结构异常的中介或团伙
```

也可以增加时间维度：

```text
时间窗口内团伙形成速度
短期集中申请
同团伙申请爆发
逾期在团伙内的传播趋势
```
