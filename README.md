# 二度关系风险图谱

本项目用于上传 CSV / Excel 贷款明细，并按「收货人手机号 = 中介」「借款人手机号 = 借款人」构建二部关系图，计算二度风险指标，并在前端页面展示概览指标、Top 列表和关系图谱。

## 项目目标

- 支持上传 CSV / Excel 文件
- 使用 `收货人手机号` 作为中介节点
- 使用 `借款人手机号` 作为借款人节点
- 每一行贷款记录作为一条业务记录
- 构建「中介 - 借款人」二部图
- 计算二度中介数、二度借款人数、二度借款逾期率
- 基于共享借款人、设备、IP、地址簇识别疑似中介团伙
- 支持在页面选择社区算法：连通分量、Louvain、Leiden
- 生成社区大小、社区坏账率、中心性等团伙特征
- 在前端页面展示指标、Top 列表和关系图谱

## 核心指标

### second_degree_agents

二度中介数。

对某个中介：

1. 找到它的一度借款人
2. 通过这些借款人反查其他中介
3. 去掉自己
4. 去重计数

### second_degree_borrowers

二度借款人数。

对某个借款人：

1. 找到它的一度中介
2. 通过这些中介反查其他借款人
3. 去掉自己
4. 去重计数

### second_degree_overdue_rate

二度借款逾期率。

对某个借款人：

1. 找到它的二度借款人集合
2. 取这些二度借款人的所有贷款记录
3. 使用逾期笔数 / 总贷款笔数计算逾期率

当前口径是：

```text
二度借款人中 final_result=30 的逾期贷款笔数 / 二度借款人中 final_result=30 的总贷款笔数
```

不是：

```text
逾期人数 / 二度借款人数
```

## 中介团伙识别

页面提供三种社区识别方式，默认使用「连通分量」。

### 方式一：连通分量

在「中介 - 借款人 - 设备 - IP - 地址簇」多关系上构建过滤后的中介关系图：

```text
中介（收货人手机号）通过共享借款人、设备、IP、地址簇关联。

共享User = 两个中介共同关联过的去重 app_user_id 数
共享Device = 两个中介各自名下直接申请记录中出现过的去重 device_id 集合交集数量
共享IP = 两个中介各自名下直接申请记录中出现过的去重 ip 集合交集数量
共享Addr = 两个中介各自名下直接申请记录中出现过的去重 addr_cluster_id 集合交集数量
```

也就是说，设备 / IP / 地址簇共享直接比较 `A1` 自己名下申请记录的属性集合与 `A2` 自己名下申请记录的属性集合；不要求两个中介必须共享借款人，也不会把某个借款人在其他中介名下的历史设备、IP、地址簇反算给 `A1` 或 `A2`。

连边过滤规则会根据首复贷筛选状态变化。

如果首复贷选择的是某一个具体取值，例如仅首贷或仅复贷：

```text
共享Device >= 1
OR 共享User >= 1
OR (共享IP >= 2 AND IP人数 <= 20)
OR 共享Addr >= 1
```

如果首复贷选择的是全部数据：

```text
共享Device >= 1
OR 共享User >= 2
OR (共享IP >= 2 AND IP人数 <= 50)
OR 共享Addr >= 1
```

其中 `IP人数` 是该 IP 关联过的去重 `app_user_id` 数。满足过滤规则后，再取这张中介关系图的连通分量作为疑似中介团伙：

```text
中介 A -- 中介 B -- 中介 C

即使 A 和 C 没有直接连边，只要它们能通过 B 连通，A、B、C 就属于同一个疑似团伙。
```

当前默认规则：

```text
至少 2 个中介构成一个疑似团伙
```

### 方式二：Louvain

先用多种关系构建用户同构图：

```text
User - Agent
User - Device
User - IP
User - AddressCluster
```

如果两个用户共享同一个中介、设备、IP 或地址簇，就在用户图里连边；多种关系会累加为更高权重。随后在加权用户图上运行 Louvain 社区发现，宽表中的 `社区id(User)`、`社区规模(User)`、`社区坏账率(User)` 等字段会按用户社区计算。

### 方式三：Leiden

服务版使用 `igraph + leidenalg` 跑标准 Leiden。它和 Louvain 都在同一套加权同构图上运行，字段仍按 `(User)` / `(Agent)` 后缀区分视角。

注意：`snapshot.html` 离线快照仍保留轻量算法，便于单文件打开；工业版 Louvain / Leiden 只在 `server.py` 服务端生效。

## 团伙特征

每个疑似中介团伙会生成以下特征：

- `agent_count`：团伙中介数
- `borrower_count`：团伙覆盖的去重借款人数
- `loan_count`：团伙覆盖借款人的贷款笔数
- `overdue_loan_count`：团伙覆盖借款人的逾期贷款笔数
- `bad_debt_rate`：团伙坏账率，当前口径为 final_result=30 的逾期贷款笔数 / final_result=30 的总贷款笔数
- `edge_count`：中介投影图中的团伙内部边数
- `density`：团伙网络密度，内部边数 / 理论最大边数
- `shared_borrower_count`：被团伙内至少两个中介共享的借款人数
- `avg_shared_borrowers_per_edge`：每条中介关联边平均共享借款人数
- `avg_degree`：团伙内中介平均度
- `max_degree`：团伙内中介最大度
- `risk_score`：规则风险分，用于排序展示
- `top_agents`：团伙内中心中介列表

中心中介特征包括：

- `degree`：该中介在团伙内直接连接的其他中介数
- `weighted_degree`：按共享借款人数加权后的度
- `degree_centrality`：度中心性
- `betweenness_centrality`：介数中心性，用于识别桥接型中介
- `first_degree_borrowers`：该中介直接关联的借款人数

当前 `risk_score` 是解释性规则分，不是训练得到的机器学习分数，主要用于原型阶段排序。

Agent 社区风险分，也就是连通分量社区和 `(Agent)` 社区，计算公式为：

```text
社区风险分
= min(
    100,
    round(
      社区坏账率 * 55
      + 社区网络密度 * 20
      + min(共享借款人占比, 1) * 15
      + min(社区中介数 / 10, 1) * 10,
      2
    )
  )
```

其中：

```text
社区网络密度 = 社区内部实际中介边数 / (社区中介数 * (社区中介数 - 1) / 2)
共享借款人占比 = 社区共享借款人数 / 社区借款人数
```

User 社区风险分，也就是 Louvain / Leiden 通用社区字段，计算公式为：

```text
社区风险分
= min(
    100,
    round(
      社区坏账率 * 55
      + 用户网络密度 * 15
      + 社区设备密度 * 10
      + 社区ip密度 * 10
      + 社区地址密度 * 10,
      2
    )
  )
```

其中：

```text
用户网络密度 = 用户社区内部实际用户边数 / (社区用户数 * (社区用户数 - 1) / 2)
社区设备密度 = 社区内部两两共享设备形成的用户边数 / 理论最大用户边数
社区ip密度 = 社区内部两两共享 IP 形成的用户边数 / 理论最大用户边数
社区地址密度 = 社区内部两两共享地址簇形成的用户边数 / 理论最大用户边数
```

## 实现过程

### 1. 搭建本地 Python HTTP 服务

项目使用 Python 标准库 `http.server` 实现接口和静态资源服务，不强依赖 Flask / FastAPI。

主要文件：

```text
server.py
```

提供接口：

```text
POST /api/analyze   上传文件并分析
POST /api/graph     查询某个中心节点的图谱
POST /api/search    搜索手机号节点
```

### 2. 实现文件上传解析

前端把 CSV / Excel 文件转成 base64 发送给后端。

后端根据文件后缀解析：

```text
.csv   使用 csv.DictReader
.xlsx  使用 openpyxl
```

当前不支持旧版 `.xls`，建议另存为 `.xlsx` 或 CSV 后上传。

### 3. 清洗字段和手机号

主要处理：

- 去掉表头 BOM
- 手机号空值过滤
- 修正 Excel 数字手机号可能出现的 `.0`
- 清理手机号里的非数字字符
- 跳过中介手机号或借款人手机号为空的行

### 4. 构建二部图关系

当前没有使用 `networkx`，而是用字典和集合实现，结构更直接。

核心结构：

```python
agent_to_borrowers
borrower_to_agents
loans_by_borrower
```

含义：

```text
agent_to_borrowers:
中介 -> 关联借款人集合

borrower_to_agents:
借款人 -> 关联中介集合

loans_by_borrower:
借款人 -> 贷款记录列表
```

### 5. 计算指标

核心逻辑位于 `RiskGraph` 类中：

```python
second_degree_agents()
second_degree_borrowers()
second_degree_overdue_rate()
agent_metrics()
borrower_metrics()
```

### 6. 前端展示

主要文件：

```text
static/index.html
static/app.js
static/styles.css
```

页面功能：

- 上传 CSV / Excel
- 拖拽上传
- 选择逾期口径
- 选择查询中心：借款人 / 中介
- 输入手机号查询图谱
- 展示概览指标
- 展示中心节点指标
- 展示疑似中介团伙 Top
- 展示二度中介 Top
- 展示二度借款逾期率 Top

- 支持导出特征 CSV / Excel 兼容 `.xls`
- 使用 SVG 渲染关系图谱

## 页面关系图谱展示逻辑

页面上的关系图谱是一个从中心节点向外展开的可视化视图，不是把全量多关系图一次性画出来。

查询中心可以是借款人，也可以是中介。不同中心类型的展开逻辑如下。

### 中心节点为借款人

如果中心节点是某个 `app_user_id`，图谱会按以下路径展开：

```text
中心借款人 -> 一度中介 -> 二度借款人
```

具体过程：

1. 先加入中心借款人节点。
2. 找到该借款人直接关联过的中介，作为一度中介节点。
3. 画出 `中介 - 借款人` 边。
4. 通过这些一度中介，找到这些中介关联过的其他借款人，作为二度借款人节点。
5. 对已经画出的 `中介 - 借款人` 业务关系，补充设备、IP、地址簇上下文节点。

上下文节点的展示规则：

```text
借款人 -> 设备
借款人 -> IP
中介 -> 地址簇
```

### 中心节点为中介

如果中心节点是某个 `consigneeMobileId`，图谱会按以下路径展开：

```text
中心中介 -> 一度借款人 -> 二度中介
```

具体过程：

1. 先加入中心中介节点。
2. 找到该中介直接关联过的借款人，作为一度借款人节点。
3. 画出 `中介 - 借款人` 边。
4. 通过这些一度借款人，找到这些借款人关联过的其他中介，作为二度中介节点。
5. 对已经画出的 `中介 - 借款人` 业务关系，补充设备、IP、地址簇上下文节点。

上下文节点的展示规则同样是：

```text
借款人 -> 设备
借款人 -> IP
中介 -> 地址簇
```

### 当前不会自动展开的关系

当前图谱展示层只沿 `中介 - 借款人` 的一度、二度关系向外展开，并补充已经展示关系上的设备、IP、地址簇上下文。

因此，以下共享关系会用于宽表特征和社区计算，但不会在页面图谱中自动展开成更多节点：

```text
两个借款人共享同一个设备
两个借款人共享同一个 IP
两个借款人共享同一个地址簇
两个中介共享同一个设备
两个中介共享同一个 IP
两个中介共享同一个地址簇
```

也就是说，如果某个用户和其他用户同属一个 `addr_cluster_id`，当前页面只会展示当前已命中关系上的地址簇节点，不会继续沿该地址簇把同簇的其他用户或中介全部展开出来。

这些关系仍然会进入以下计算：

```text
地址簇关联人数
地址簇关联收货人手机号数
共享地址簇图
多关系中介投影图
连通分量关系图
Louvain / Leiden 用户同构图
社区地址密度
社区风险分
```

### 展示图谱与计算图的区别

可以把当前系统理解为两层图：

```text
页面展示图谱 = 中心节点的一度/二度中介-借款人关系 + 设备/IP/地址簇上下文
特征计算图 = 共享用户、共享设备、共享 IP、共享地址簇形成的投影图 / 同构图
```

因此，页面上没有展开某条设备、IP、地址簇关系，不代表这条关系没有参与宽表特征或社区计算。

## 复杂查询模块

导航栏中的「复杂查询」用于在已上传数据上做批量校验。数据只需要在「图谱分析」页上传一次，切换到「复杂查询」页后会复用当前浏览器页面中缓存的上传数据；服务版同时通过 `dataset_id` 在后端缓存本次上传数据。

查询条件包含三类下拉筛选：

```text
首复贷 reloan_flag
是否退货 return_flag
借款结果 final_result
```

多值输入框支持以下实体字段：

```text
app_user_id
consigneeMobileId
device_id
ip
addr_cluster_id
```

多个值用 `|` 分隔。比如：

```text
49.80.49.85|223.104.40.12|117.136.66.3
```

命中逻辑为：

```text
先应用 reloan_flag / return_flag / final_result 下拉筛选
再判断任意一个多值输入框是否命中对应字段
只要任意字段命中，就返回该原始行
```

如果没有输入任何实体值，则只按下拉筛选返回原始行。

命中概览会统计：

```text
命中行数
命中 loan_task_id 数
用户数 app_user_id
收货人手机号数 consigneeMobileId
设备数 device_id
IP数 ip
地址簇数 addr_cluster_id
原始地址数 consigneeAddr
预处理地址数 receiverAddr
```

页面还会展示每类输入值的：

```text
输入值数量
命中值数量
未命中值数量
未命中的具体值
```

命中明细表返回的是上传文件的原始列和原始行，支持每页 `20 / 50 / 100` 条分页展示。

### 加工特征计算口径

以下说明以宽表中的当前行为准：

```text
U = 当前行 app_user_id / 借款人
A = 当前行 consigneeMobileId / 收货人手机号 / 中介
C = 当前行 addr_cluster_id / 地址簇
D = 当前行 device_id / 设备
P = 当前行 ip
```

通用口径：

- `申请笔数` 类字段：统计筛选后的全量申请行
- `贷款笔数`、`坏账率` 类字段：只统计 `final_result=30` 的通过样本
- `坏账率`：通过样本中逾期笔数 / 通过样本总笔数
- 所有 `坏账率` 类字段的逾期判断都跟随页面“逾期口径”下拉框，例如选择 `fpd1` 就按 `fpd1` 判断逾期，选择 `cpd7` 就按 `cpd7` 判断逾期
- `关联人数`、`关联设备数`、`关联ip数`、`关联地址簇数`：均为去重计数
- 页面上的首复贷、是否退货、逾期口径筛选会影响所有加工结果
- 概览中的 `上传原始行数` 是文件解析出的总行数，`参与分析行数` 是应用首复贷/退货筛选后的行数

### 中介投影图特征：共享用户口径

先构建「共享用户」中介投影图：

```text
如果两个中介共享同一个 app_user_id，则两个中介连边
边权 = 两个中介共享的 app_user_id 数量
```

当前行取中介 `A` 在这张图上的特征：

```text
节点度数(共享用户) = A 直接连接的其他中介数量
加权度数(共享用户) = Σ weight(A, B)，B 为 A 在共享用户图上的所有相邻中介
PageRank分数(共享用户) = 在共享用户中介投影图上的 PageRank
中介中心性(共享用户) = A 在共享用户中介投影图上的介数中心性
连通子图大小(共享用户) = A 所在连通分量中的中介数量
社区id(共享用户) = A 所在共享用户连通分量 ID
```

其中：

```text
weight(A, B) = A 与 B 共同关联过的去重 app_user_id 数
```

这里的“边权”就是图中两点之间那条边本身的权重；“加权度数”就是把当前节点所有相邻边的边权直接求和。

其中 `中介中心性(共享用户)` 是介数中心性，衡量 `A` 是否位于其他中介之间的最短路径上；它不是“两个借款人通过一个中介相连”的二部图桥接数。

计算公式：

```text
BC(A) = Σ σ(s,t | A) / σ(s,t)
中介中心性(共享用户) = BC(A) / ((n - 1) * (n - 2))
```

其中：

```text
s、t = 共享用户中介投影图中除 A 以外的两个中介节点
σ(s,t) = s 到 t 的最短路径条数
σ(s,t | A) = s 到 t 的最短路径中经过 A 的条数
n = A 所在共享用户连通分量中的中介数量
```

当前实现按无权最短路径计算，也就是只看路径经过多少条边，不把共享用户边权作为路径距离。若单个社区中介数超过 80，为避免精确计算过慢，该社区的中介中心性会置为 0。

### 中介投影图特征：连通分量关系口径

连通分量不再只看共享用户，而是先构建一张过滤后的中介关系图。

两个中介 `A1`、`A2` 的关系计数如下：

```text
共享User = A1 关联借款人与 A2 关联借款人的交集数量
共享Device = A1 名下直接申请记录出现过的设备集合，与 A2 名下直接申请记录出现过的设备集合的交集数量
共享IP = A1 名下直接申请记录出现过的 IP 集合，与 A2 名下直接申请记录出现过的 IP 集合的交集数量
共享Addr = A1 名下直接申请记录出现过的地址簇集合，与 A2 名下直接申请记录出现过的地址簇集合的交集数量
IP人数 = 某个 IP 关联过的去重 app_user_id 数
```

设备 / IP / 地址簇的比较不要求两个中介必须存在共同借款人。若某个借款人还关联了 `A3`，该借款人在 `A3` 名下出现过的设备、IP、地址簇不会计入 `A1` 与 `A2` 的共享关系。

过滤连边条件会根据首复贷筛选状态变化。

如果首复贷选择的是某一个具体取值，例如仅首贷或仅复贷：

```text
共享Device >= 1
OR 共享User >= 1
OR (共享IP >= 2 AND IP人数 <= 20)
OR 共享Addr >= 1
```

如果首复贷选择的是全部数据：

```text
共享Device >= 1
OR 共享User >= 2
OR (共享IP >= 2 AND IP人数 <= 50)
OR 共享Addr >= 1
```

当前行取中介 `A` 在过滤后关系图上的特征：

```text
节点度数(连通分量关系) = A 在过滤后关系图上直接连接的其他中介数量
加权度数(连通分量关系) = Σ weight(A, B)，B 为 A 在过滤后关系图上的所有相邻中介
PageRank分数(连通分量关系) = A 在过滤后关系图上的 PageRank
中介中心性(连通分量关系) = A 在过滤后关系图上的介数中心性
连通子图大小(连通分量关系) = A 所在过滤后中介连通分量中的中介数量
社区id(连通分量关系) = A 所在过滤后中介连通分量 ID
```

其中过滤后关系图的边权为：

```text
weight(A, B)
= 1.0 * 共享User(A, B)
+ 1.2 * 共享Device(A, B)
+ 0.8 * 共享IP(A, B)
+ 1.0 * 共享Addr(A, B)
```

注意：这里的 `共享IP(A, B)` 只统计满足当前 IP 人数阈值的 IP。
这里的“加权度数”同样就是把 `A` 与所有相邻中介之间的边权求和。

### 中介投影图特征：单类关系口径

分别构建三张中介投影图：

```text
共享设备图：两个中介各自名下直接申请记录出现过相同 device_id 则连边，边权 = 共享 device_id 数
共享IP图：两个中介各自名下直接申请记录出现过相同 ip 则连边，边权 = 共享 ip 数
共享地址簇图：两个中介各自名下直接申请记录出现过相同 addr_cluster_id 则连边，边权 = 共享 addr_cluster_id 数
```

当前行取中介 `A` 在对应图上的特征：

```text
节点度数(共享设备) = A 在共享设备图上直接连接的其他中介数量
加权度数(共享设备) = Σ weight_device(A, B)，weight_device 为 A 与 B 各自名下直接申请记录的去重 device_id 集合交集数量
节点度数(共享IP) = A 在共享 IP 图上直接连接的其他中介数量
加权度数(共享IP) = Σ weight_ip(A, B)，weight_ip 为 A 与 B 各自名下直接申请记录的去重 ip 集合交集数量
节点度数(共享地址簇) = A 在共享地址簇图上直接连接的其他中介数量
加权度数(共享地址簇) = Σ weight_addr(A, B)，weight_addr 为 A 与 B 各自名下直接申请记录的去重 addr_cluster_id 集合交集数量
```

也就是说，这里的边权分别来自共同设备数、共同 IP 数、共同地址簇数，节点的加权度数就是把这些边权加总。

### 中介投影图特征：多关系口径

多关系中介投影图同时使用共享用户、共享设备、共享 IP、共享地址簇：

```text
Agent -- 共享用户 -- Agent
Agent -- 共享设备 -- Agent
Agent -- 共享IP -- Agent
Agent -- 共享地址簇 -- Agent
```

其中共享设备、共享 IP、共享地址簇均使用中介名下直接申请记录口径，不要求两个中介必须共享借款人，也不使用借款人在其他中介下的全历史属性。

当前边权：

```text
多关系边权
= 1.0 * 共享用户数
+ 1.2 * 共享设备数
+ 0.8 * 共享 IP 数
+ 1.0 * 共享地址簇数
```

过滤后的连通分量关系图使用同一套边权，但 `共享 IP 数` 只统计满足当前 IP 人数阈值的 IP：

```text
仅首贷或复贷：IP人数 <= 20
全部数据：IP人数 <= 50
```

User-User 同构图边权为：

```text
用户边权
= 1.0 * 共享中介数
+ 1.2 * 共享设备数
+ 0.8 * 共享 IP 数
+ 1.0 * 共享地址簇数
```

当前行取中介 `A` 在多关系图上的特征：

```text
节点度数(多关系) = A 在多关系图上直接连接的其他中介数量
加权度数(多关系) = Σ weight_multi(A, B)，B 为 A 在多关系图上的所有相邻中介
PageRank分数(多关系) = A 在多关系图上的 PageRank
中介中心性(多关系) = A 在多关系图上的介数中心性
连通子图大小(多关系) = A 所在多关系连通分量中的中介数量
社区id(多关系) = A 所在多关系连通分量 ID
```

其中：

```text
weight_multi(A, B)
= 1.0 * 共享用户数(A, B)
+ 1.2 * 共享设备数(A, B)
+ 0.8 * 共享 IP 数(A, B)
+ 1.0 * 共享地址簇数(A, B)
```

### PageRank 计算公式

PageRank 用在共享用户图、连通分量关系图、多关系 Agent 图和 `(Agent)` Louvain / Leiden 图上。当前实现是加权 PageRank，阻尼系数固定为：

```text
d = 0.85
```

第 `t+1` 轮迭代公式为：

```text
PR_{t+1}(v)
= (1 - d) / N
+ d * Dangling_t / N
+ d * Σ_{u∈In(v)} PR_t(u) * w(u,v) / Σ_{x∈Out(u)} w(u,x)
```

其中：

```text
N = 图中节点总数
w(u,v) = u 到 v 的边权
Out(u) = u 的邻居集合
In(v) = 指向 v 的邻居集合；无向图中等价于 v 的邻居集合
Dangling_t = 第 t 轮所有无出边节点的 PageRank 分数之和
```

初始化：

```text
PR_0(v) = 1 / N
```

当前实现固定迭代 40 轮。

### 社区特征

页面选择不同社区算法时，通用社区字段会跟随算法变化：

```text
连通分量：社区 = 过滤后中介关系图中的中介连通分量
Louvain：社区 = 用户同构图上的 igraph Louvain 用户社区
Leiden：社区 = 用户同构图上的 leidenalg 标准 Leiden 用户社区
```

当页面选择 Louvain / Leiden 时，宽表还会额外生成一套 `(Agent)` 后缀字段。它们来自未过滤的 Agent-Agent 多关系加权图：

```text
Agent -- 共享用户 -- Agent
Agent -- 共享设备 -- Agent
Agent -- 共享IP -- Agent
Agent -- 共享地址簇 -- Agent
```

Agent 图不使用连通分量过滤规则，直接用多关系边权跑 Louvain / Leiden。

这里的 Agent 图同样使用中介名下直接申请记录口径计算共享设备、共享 IP 和共享地址簇，不要求两个中介必须共享借款人。

对应字段：

```text
社区id(Agent) / 社区id(User) = 当前行命中的社区 ID
社区规模(Agent) = 连通分量或 Agent Louvain / Leiden 社区中的中介数
社区规模(User) = Louvain / Leiden 用户社区中的借款人数
社区借款人数(Agent/User) = 社区覆盖的去重 app_user_id 数
社区贷款笔数(Agent/User) = 社区覆盖记录中 final_result=30 的笔数
社区坏账率(Agent/User) = 社区覆盖记录中 final_result=30 且逾期的笔数 / final_result=30 的笔数
社区设备密度(Agent/User) = 社区内部节点两两共享设备形成的边数 / 理论最大边数
社区ip密度(Agent/User) = 社区内部节点两两共享 IP 形成的边数 / 理论最大边数
社区地址密度(Agent/User) = 社区内部节点两两共享地址簇形成的边数 / 理论最大边数
社区共享借款人数(Agent) = 社区内被至少两个中介共享的借款人数
社区共享借款人数(User) = 当前为 0
社区风险分(Agent/User) = 由坏账率、网络密度、设备/IP/地址密度等组合得到的解释性排序分，不是模型概率
```

字段输出规则：

```text
连通分量：输出社区字段后缀为 (Agent)
Louvain / Leiden：输出用户社区字段后缀为 (User)，并额外输出 Agent-Agent 多关系图字段后缀为 (Agent)
```

## 离线快照版

如果不想通过 `http://127.0.0.1:8000` 访问，可以直接打开项目根目录下的：

```text
snapshot.html
```

这个文件是一个单文件离线页面，包含页面样式、计算逻辑和图谱渲染逻辑。它不依赖 `server.py`，后续修改服务端或前端源码不会影响这个快照文件。

使用方式：

1. 在文件管理器里双击 `snapshot.html`
2. 浏览器打开后上传 CSV 或 `.xlsx`
3. 在页面内查看概览指标、中心指标、Top 列表和关系图谱

适合场景：

- 发给同事在其他设备上直接打开
- 保留某个版本的演示页面
- 在没有 Python 环境的机器上查看图谱
- 避免后续代码修改影响演示版本

离线快照版注意事项：

- 支持 CSV 和常见 `.xlsx`
- 不支持旧版 `.xls`
- `.xlsx` 解析依赖浏览器本地解压能力，建议使用最新版 Chrome、Edge 或 Safari
- 所有计算都在浏览器内完成，上传文件不会发送到服务器
- 数据量特别大时，浏览器页面可能会卡顿，建议先用 CSV 或分批分析

## 局域网访问

当前默认只允许本机访问：

```python
HOST = "127.0.0.1"
```

如果想让同一局域网里的其他设备访问，把 `server.py` 中的配置改为：

```python
HOST = "0.0.0.0"
```

然后重新启动：

```bash
python3 server.py
```

查看本机局域网 IP：

```bash
ipconfig getifaddr en0
```

假设输出是：

```text
192.168.1.23
```

其他设备访问：

```text
http://192.168.1.23:8000
```

前提是其他设备和你的电脑在同一个 Wi-Fi / 局域网内。

常见阻碍：

- macOS 防火墙可能会询问是否允许 Python 接收入站连接，需要允许
- 公司或校园 Wi-Fi 可能开启设备隔离，同一 Wi-Fi 下也无法互相访问
- 如果端口 `8000` 被占用，可以用 `PORT=8080 python3 server.py` 换端口

## 注意事项

### 数据隐私

上传数据里包含手机号和风控字段，属于敏感数据。当前页面展示时手机号做了简单脱敏，但后端内存里仍然保存原始手机号。

不建议直接暴露到公网。

### 文件格式

当前支持：

```text
.csv
.xlsx
.xlsm
.xltx
.xltm
```

当前不支持：

```text
.xls
```

### 字段名要一致

尤其是这两个字段：

```text
收货人手机号
借款人手机号
```

如果字段名不同，需要在代码里增加字段映射。

### 重复关系会去重

同一个中介和同一个借款人出现多笔贷款时：

```text
关系边只算一条
贷款记录仍按多笔保留
```

因此：

- 二度人数 / 中介数：按去重节点算
- 逾期率：按贷款笔数算

### 图谱展示可能截断

为了避免前端卡顿，二度节点默认限制展示一部分。指标本身仍按完整集合计算，图谱展示只是可视化视图。

### 当前数据保存在内存

服务重启后，上传的数据会消失。当前版本适合本地分析和演示，不是持久化系统。

## 后续增强方向

- 增加字段映射页面，允许用户选择哪一列是中介、借款人、逾期字段
- 支持旧版 `.xls`
- 增加结果导出 CSV / Excel
- 增加团伙识别阈值配置，例如最小共享借款人数、最小团伙中介数
- 增加团伙详情页，展示团伙内部中介投影图和借款人明细
- 增加按渠道 `app_key(渠道)` 筛选
- 增加按申请时间 `apply_time` 筛选
- 增加风险规则，例如二度逾期率超过阈值自动标红
- 增加借款人详情面板，展示贷款明细
- 接入 Neo4j，支持大规模图查询
- 增加登录认证，避免敏感数据被他人访问
- 支持部署到服务器，并开启 HTTPS
- 增加手机号哈希化，降低隐私泄露风险

## 适合的使用场景

当前版本适合：

```text
本地上传数据
快速验证二度关系指标
做风控分析原型
给团队演示关系图谱
探索中介和借款人的关联风险
```

如果要进入生产环境，建议补上：

```text
权限控制
数据脱敏
日志审计
文件大小限制
异常文件处理
数据持久化
任务异步计算
```
